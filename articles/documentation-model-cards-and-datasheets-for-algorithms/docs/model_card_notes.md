# Model Card Notes

A model card should document:

- model name, version, owner, and date;
- model type, inputs, outputs, dependencies, and decision role;
- intended users and intended use;
- prohibited or unsupported uses;
- training and evaluation data summaries;
- metrics, baselines, subgroup performance, calibration, and robustness;
- ethical considerations, failure modes, and known limitations;
- governance owners, monitoring, incident response, and maintenance cadence.

A model card should tell readers when the model is unsuitable, not only when it performs well.
